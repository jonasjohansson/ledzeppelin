# 3D Mapping (2D/3D mode) Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Give fixtures/points real XYZ and project the existing 2D composition onto the 3D-arranged rig via a fixed projection camera, so a 2D sweep travels through an arc with real foreshortening — with 2D as the flat-front special case and every existing show unchanged.

**Architecture:** Points carry `[x,y,z]` (missing z ⇒ 0). A new pure `project3d.js` turns world XYZ + a camera into canvas UV. `pipeline.js`'s `fixtureSampleUVs` resamples a fixture's polyline by **3D arc-length** then **projects each LED** to UV; 2D fixtures keep the existing 2D path byte-for-byte. The output/daemon path is untouched — 3D is authoring-only. See the design: `docs/plans/2026-07-01-3d-mapping-design.md`.

**Tech Stack:** Vanilla ESM, no build step. Hand-rolled projection (zero new runtime deps). Tests: `node --test "test/*.test.js"` (`node:test` + `node:assert/strict`).

**Conventions:** Test files `test/<name>.test.js`, `import { test } from 'node:test'; import assert from 'node:assert/strict';`, import from `../src/model/…`. Commit after every green step. Run the FULL suite before each commit (297 tests must stay green).

**Step 3 — minimal impl:**

```js
// src/model/project3d.js
// Hand-rolled camera projection (zero deps). World coords share today's normalized
// canvas space: x,y ∈ 0..1 across the canvas, z in the same unit scale (z=0 = the
// canvas plane). 2D mode uses a FLAT camera that drops z, reproducing today exactly.

// A camera config: { mode: 'flat' | 'ortho' | 'perspective', pos, target, up, fov, orthoHeight, aspect }.
export function flatCamera() { return { mode: 'flat' }; }

export function project(P, cam) {
  if (!cam || cam.mode === 'flat') return [P[0], P[1]]; // exact — no float drift for 2D regression
  return projectFramed(P, cam);
}
```
(`projectFramed` is added in Task 2; for now it can `throw`.)

**Step 4 — run, expect PASS.**

### Task 2: `project3d.js` — perspective + ortho camera (lookAt)

**Files:** Modify `src/model/project3d.js`; Test `test/project3d.test.js`.

**Step 1 — failing tests** (pin the math with hand-computed values):

```js
import { perspectiveCamera, project } from '../src/model/project3d.js';

test('perspective: a point on the camera axis lands at centre UV', () => {
  // camera at (0.5, 0.5, 1) looking toward -z at the canvas centre; 90° fov, square.
  const cam = perspectiveCamera({ pos: [0.5, 0.5, 1], target: [0.5, 0.5, 0], fov: 90, aspect: 1 });
  const [u, v] = project([0.5, 0.5, 0], cam);
  assert.ok(Math.abs(u - 0.5) < 1e-9 && Math.abs(v - 0.5) < 1e-9);
});

test('perspective: equal offsets at greater depth project closer to centre (foreshortening)', () => {
  const cam = perspectiveCamera({ pos: [0.5, 0.5, 1], target: [0.5, 0.5, 0], fov: 90, aspect: 1 });
  const near = project([0.7, 0.5, 0], cam);   // depth 1 from camera
  const far  = project([0.7, 0.5, -1], cam);  // depth 2 from camera
  assert.ok(Math.abs(far[0] - 0.5) < Math.abs(near[0] - 0.5)); // farther → nearer centre
});
```

**Step 2 — run, expect FAIL.**

**Step 3 — impl** `perspectiveCamera(cfg)`, `orthoCamera(cfg)`, and `projectFramed(P, cam)` using a hand-rolled basis (forward = normalize(target−pos), right = normalize(cross(forward, up)), trueUp = cross(right, forward)); camera-space `(cx,cy,cz)` via dot products; perspective divide `x = cx / (cz·tan(fov/2)·aspect)`, `y = cy / (cz·tan(fov/2))`; UV `= [(x+1)/2, (1−y)/2]`. Guard `cz ≤ 0` (behind camera) → return the point flagged/clamped. Include small vec helpers (sub/dot/cross/normalize).

**Step 4 — run, expect PASS.** **Step 5 — commit.**

### Phase-2 prerequisites (from the Phase-1 whole-feature review)

Latent sharp edges Phase-1 code never triggers but Phase-2 *editing* will. Resolve first, before wiring the viewport/editor:

1. **Behind-camera LEDs must blacken.** `projectFramed` returns `[NaN, NaN]` for points at/behind the camera; those UVs currently reach `engine/sampler.js` and sample undefined (isolated per-LED, no buffer corruption — but not black). Honour the "drop them" intent: substitute an out-of-range sentinel (`[-1,-1]`, which the sampler's bounds check already blackens) in `pipeline.js`, or add `isnan`/`!(c==c)` handling in the sampler shader.
2. **`samplePoints3D` must tolerate mixed 2-/3-tuples** — a 2-tuple makes `dz = NaN` → all-`null` output. Default `p[2] ?? 0`.
3. **`normPts` should promote a whole polyline to 3D once any vertex has z** (avoid per-point dimensionality splits, e.g. an inserted 2-tuple midpoint in a 3D run). Pairs with (2).

Optional: an `aspect: 2` projection test (all current tests use `aspect: 1`); deeper `cameraFromView3d` field validation once the UI builds cameras incrementally.
